Representational State Transfer

바쁜 개발자들을 위한 REST 논문 요약

REST는 Representational State Transfer의 줄임말로, 웹을 위한 네트워크 기반 아키텍처 스타일이다. REST는 Roy T. Fielding이 그의 박사학위 논문 “Architectural Styles and the Design of Network-based Software Architectures” 에서 처음 소개하였다.

아하.

REST 아키텍처 스타일

REST 아키텍처에 대한 설명은 온갖 블로그에서 서로 다르게 해석하고 있다.

작성자의 역량 + 독자의 역량이 오해를 재생산하는가 보다.

그러니 지금 이 정리를 본 누군가도 역시 오해를 얻어가리라 믿는다. 나 때문이든 당신 때문이든.

‘바쁜 개발자들을 위한 REST 논문 요약’을 요약 + WhatIsRest.com + 내 생각을 적어보자.

REST 제약 조건

Client-Server

  • 보통 생각하는 서버-클라이언트를 의미
  • 서로의 관심사를 분리하여 각각의 로직의 독립적인 진화를 지원
  • 여기서 의문이 들었던 게, Client-Server라는 게 과연 제약조건이라고 할 수 있는 건가? 그러니까, 제약 조건을 어기는 게 과연 가능한가?
  • 이 제약조건은 클라이언트와 서버의 역할을 명확하게 구분한다 정도로 생각하면 될 것 같다

Stateless

  • 모든 요청은 필요한 모든 정보를 담고 있어야한다
  • 요청 하나만 봐도 바로 뭔지 알 수 있으므로 가시성이 확보된다
  • 그렇다면 인증을 위해 쿠키를 심고 세션에서 필요한 정보를 가져오는 경우, 이는 stateless인가 아닌가?
  • API 문외한 입장에서 가장 이해가 안됐던 부분 중 하나인데, 난 서버에 세션 정보를 저장하는 방법 말고는 몰랐기 때문이다
  • REST에서는 각각의 요청에 서버가 필요한 정보가 모두 담겨야 한다

Cache

  • 모든 서버 응답은 캐시 가능한지 그렇지 아닌지 알 수 있어야한다
  • 그리고 캐시를 고려한 설계

Uniform Interface

  • 구성요소(클라이언트, 서버 등) 사이의 인터페이스는 균일(uniform)해야한다
    • 이것만으로는 무슨 말인지 모르겠다

  • WhatIsREST.com의 내용을 구글 번역으로 돌려보면
    소비자는 많은 소비자와 서비스에서 표준화 된 방법, 미디어 유형 및 공통 리소스 식별자 구문을 통해 서비스 기능에 액세스합니다
    라고 나오는데, 미디어 유형이나 리소스 식별자 등을 의미하는 문법이 구성요소나 호환 시스템 간 동일해야 한다

  • REST는 네가지 Interface 제약조건으로 정의된다

    • identification of resources
      • 리소스는 저장된 특정 객체를 의미하는 게 아니고 하나의 개념으로 이해해야 한다
      • (구글 번역된 논문을 살펴보면) 이 자원의 추상적 정의는 웹 아키텍처의 핵심 기능을 가능하게합니다
        • 유형이나 구현에 따라 인위적으로 구별하지 않고 많은 정보 소스를 포괄함으로써 보편성을 제공합니다
        • 표현에 대한 참조의 늦은 바인딩을 허용하여 요청의 특성에 따라 내용 협상을 수행 할 수 있습니다
        • 저자는 개념의 단수 표현보다는 개념을 참조 할 수 있으므로 표현이 바뀔 때마다 기존의 모든 링크를 변경할 필요가 없습니다
    • manipulation of resources through representation
      • representation의 개념은 이후에 다시 다룰텐데, 이 역시 이응준님 블로그에서 다루고 있다
      • 리소스의 표현계층(representation)을 리소스의 식별자(URL)로부터 분리한 것은 REST에서 아주 주요한 관점이다
      • RESTful 응용 프로그램은 동일한 URI에서 동일한 자원의 표현을 둘 이상 지원할 수 있다
    • self-descriptive messages
      • 요청이나 응답 메시지에는 이후 처리를 위한 모든 정보가 들어있어야 한다
      • 논문에서 이를 설명하는 문구를 보자
        REST enables intermediate processing by constraining messages to be self-descriptive: interaction is stateless between requests, standard methods and media types are used to indicate semantics and exchange information, and responses explicitly indicate cacheability.
    • hypermedia as the engine of application state
      • = HATEOAS
      • 조대협님의 블로그에 언급되기를, HTTP Response에 다음 Action이나 관계되는 리소스에 대한 HTTP Link를 함께 리턴하는 것
        • 뭔가…이상한데…틀렸다고 하기는 뭣 하지만 온전히 이해가 되는 문장은 또 아니다. 다른 사람의 의견을 더 들어보자
      • What HATEOAS actually means
        • 다들 혼란스러워하는 것 같은데 직접 로이 필딩에게 물어보지 그래?
        • 로이의 문장은 명확하고, 모호하지도 않으니 그냥 읽어보길 바란다
        • 로이가 HTTP 표준의 기여자라는 것을 잘 생각해보자
        • 이 논문의 주제는 WWW가 잘 동작하고 확장성도 좋고, REST가 WWW 아키텍쳐 위에서 동작한다면, 이 역시 잘 동작하고 확장성이 좋다는 것이다
        • 4가지의 제약 조건은 결국 URI를 사용해서 MIME-typed 문서를 GET/PUT 등으로 전송하는 것 + HATEOAS
        • WWW := URI + HTTP + MIME + Hypermedia
        • Hypermedia라는 건 hypertext를 확장한 개념
        • 좀 더 정확하게 쓰자면, HATEOAS := hypermedia documents as the state of state
        • 로이의 배경을 생각하면 그는, 하이퍼링크가 없는 HTML 도큐먼트를 상상조차 못할 인물이다
        • 그가 언급한 Hypermedia란 문서 자체를 의미한다
        • 로이는 HTTP통신에 Link header를 넣어도 여전히 RESTful 할 수 있지만, 합당한 이유가 있어야 한다고 했다
          • 헤더에 넣으면 이를 처리할 수 없는 클라이언트도 있을 수 있기 때문에..라고 했는데 당시에는 LINK가 정식 표준이 아닌 때였기 때문인 것 같음
        • HATEOAS를 정확히 구현한 것은 문서안에 있는 링크다
          • representation 안이라고 명시하는 것을 보니 header는 representation이라고 표현 안 하는 듯
        • HATEOAS라는 다음 상태 변경을 위한 링크(form 링크 등)를 제공해야 한다는 것

Layered System

  • 클라이언트든 서버든 미들웨어 구성요소를 추가할 수 있는 구조
  • 하지만 Client-Server 사이에선 그 구성요소가 추가되는지, 다른 서비스와 추가로 통신하는 지 여부는 관심없다
  • 특정 서비스와 소비자 간의 상호 작용을 일관성있게 유지해야 한다

Code-On-Demand (Optional)

  • 서버가 네트워크를 통해 클라이언트에 프로그램을 전달하면 그 프로그램이 클라이언트에서 실행될 수 있어야한다. (Java applet이나 Javascript 같은 것을 말함) 다만 이 제약조건은 필수는 아니다

REST VS RESTful

REST는 2000년에 발표한 논문이다. 그리고 20년 쯤 지났다.

로이 필딩은 HTTP에 기여를 했고 웹이 얼마나 잘 설계되었는지에 대한 확신이 있었던 것 같다.

여전히 그는 틀리지 않았다고 생각한다. 웹은 여전히 잘 돌아간다.

하지만 REST가 현재의 웹을 잘 담아낼 수 있을까 의문이다.

현대의 요구사항에 맞춰 제약 조건을 조금 위반한 설계를 여전히 REST라고 부를 수 있을까?

잠깐.

당시의 웹 환경과 철학이 과연 얼마나 달라졌는가 생각을 해보자.

그동안 웹 사이트 내에 동적인 요소가 많이 추가됐고 타 서비스 간 통신도 활발해졌지만,

웹의 철학이 바뀌었는가?

웹은 그대로 있지 않았나?

RESTful?

REST와 RESTful을 동일하게 보는 시선도 분명 있지만(보통 그냥 막 섞어쓰기 때문에..),

REST는 아키텍쳐 스타일, RESTful은 REST에 기초하여 구현한 시스템을 의미한다고 생각한다.

(사실 스택오버플로우에서 그런 주장을 봤다)

위키디피아에서 설명하는 REST에는

‘Representational state transfer (REST) or RESTful Web services는’이라고 시작하는데,

REST와 RESTful Web service를 동급으로 본다.

앞서 REST의 의미를 살펴본 바 틀리지 않은 것 같다.

API는 웹인가?

(API의 개념을 좁게 보면) 웹의 중요한 요소이지만 그 자체로 온전한 웹이라고 할 수 있을까?

이게 좀 애매한데, 이건 사용방식에 딸라 달라질 것 같다.

API는 리소스나 어플리케이션의 상태를 변경하는 용도를 쓰게 될텐데,

온전한 하나의 어플리케이션으로서 혹은 그것의 상태를 변경하는 API도 있겠지만

어플리케이션의 서브 리소스로써 동작하는 API도 존재한다.

이미지와 동영상은 어떨까?

논문에서 hypertext가 아닌 hypermedia로 언급한 것처럼, 이 거대한 웹에는 여러가지 미디어가 연결되어 있다.

하지만 이를 연결하는 주체는 웹(특히 HTML)이 아닌가?

이런 이미지 등의 리소스는 HATEOAS를 제공할 의무가 있는가?

난 Cache에 대한 제약사항이 ‘모든 response는 캐시가 가능해야 한다’가 아니고 ‘캐시가 가능한 지 알 수 있어야 하고, 캐시를 고려해야 한다’라고 본다.

HATEOAS도 마찬가지의 관점에서 봐야하지 않을까 생각이 든다.

그렇다면 RESTful 서비스는 REST의 모든 제약조건을 지킨 시스템을 말하는, 모든 제약조건을 고려한 시스템을 의미하는가?

여기서 고려한이란 말은 안 지켜도 된다가 아니라, 지킬 수 없는 상황도 있다 정도로 생각하고 있다.

최대한 RESTful

간단히 말해서 RESTful하다는 말은 다섯(+1)가지의 제약 조건을 충족시킨다는 의미이다.

다섯가지의 제약조건을 충족시켰다는 것은 웹의 특성을 잘 살려 일관되고 효율적이며 확장성이 있게 만들었다는 말이다.

어떤 이는 REST API는 네가지 메소드를 사용해 통신하는 방식을 의미하고, 로이 필딩이 주장하는 디자인은 특별히 RESTful이라고 지칭한다고 말한다.

사실 말도 안되는 이야기이긴 하나, REST라는 용어가 그만큼 (부정확한 개념으로) 널리 퍼졌다는 뜻이기도 하다.

REST라는 용어를 아키텍처 스타일 보다는 API 구현물을 묘사하는데 사용하고 있다.

GET/POST/PUT/DELETE를 사용할 수 있는 API.

아키텍처 스타일일 뿐 실무에서 모든 제약조건을 따를 수도 따를 필요도 없다는 사람도 있다.

그렇다면 REST가 아닌 다른 용어를 만들어 써야한다는 사람도 있다.

나는 이제 후자에 속하는 사람이지만, REST의 정확한 개념으로는 정확한 의사소통을 하기 어려울 수도 있다고 생각한다.

REST를 단편적으로 알고 있는 갑 회사의 매니저나 면접관과 이야기 한다고 생각을 해보자.

REST에 관해서 이야기할 때는 서로의 관점을 통일할 필요가 있다고 본다.
